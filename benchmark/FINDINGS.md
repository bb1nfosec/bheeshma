# Benchmark Findings

Generated by `node benchmark/run.js` against a labeled corpus of 7 malicious +
7 benign fixtures modeled on real supply-chain attacks (Shai-Hulud / event-stream
secret theft, ua-parser-js cryptojacking, flatmap-stream DNS tunneling,
dependency-confusion reverse shell, obfuscated loader, crossenv typosquat,
async-deferred exfil) and on ordinary legitimate dependencies (dotenv config
loader, HTTPS API client, file logger, config reader, build tool that spawns a
subprocess, analytics SDK, pure util).

This is a **first, synthetic** corpus -- useful for measuring the engine's
behavior and catching regressions, not yet a claim about real-world efficacy.
Replacing it with real malicious packages (in a sandbox) is the Stage-2 follow-up.

## Detection vs false positives by enforcement threshold (current)

| Threshold | Recall (catch rate) | Precision | FP rate | F1 |
|---|---|---|---|---|
| critical-only (<30) -- current default | 29% (2/7) | 100% | 0% | 0.44 |
| **high+ (<60) -- recommended** | **71%** (5/7) | **100%** | **0%** | **0.83** |
| medium+ (<80) | 71% (5/7) | 71% | 29% | 0.71 |

### How the engine got here (each step measured, not assumed)

| Engine state | default-gate recall | high+ recall | high+ FP |
|---|---|---|---|
| As shipped (with the env-flood bug) | 43%* | 43%* | 14%* |
| After env-flood fix (artifact removed) | 0% | 29% | 0% |
| **After correlation-aware scoring** | **29%** | **71%** | **0%** |

\* The "as shipped" numbers were fake: the `ENV_ACCESS`-flood-on-spawn bug
floored every subprocess-spawning package (malicious **and** benign) to score 0,
inflating apparent detection and producing a benign-build-tool false positive.

## Headline findings

1. **The shipped default gate (`fail-level=critical`) is too permissive.** Even
   after improvements it catches 29%. **`high+` is the right default**: 71%
   recall at 100% precision / 0% false positives on this corpus. The default
   should be changed (or at minimum documented as permissive).

2. **Correlation-aware scoring was the keystone fix.** The pattern matcher
   already recognized exfil/backdoor/crypto/credential-theft/typosquat
   combinations, but those findings never affected the trust score (pure
   additive). Letting a recognized pattern cap the score into its risk band
   lifted high+ recall from 29% to 71% with no new false positives, and
   correctly left the context-aware `dotenv` exemption alone (stays LOW).

3. **Two attack classes still score LOW and are missed even at medium+:**
   - **DNS tunneling (89).** The `dnsHook` computes high-entropy / known-exfil
     indicators in signal metadata, but there is **no DNS pattern detector** to
     elevate them. Adding one is the next clear win.
   - **Obfuscated loader (87).** The `OBFUSCATION_DETECTED` signal did not fire
     in the run — the scan is scheduled via `setImmediate` and can lose the race
     with report generation (a known P1 bug), and there is no obfuscation-based
     correlation pattern.

4. **Residual medium+ false positives** are a benign build tool (spawns a
   subprocess) and an analytics SDK (env + file + one HTTPS call), both at 72-77.
   These are the genuinely hard, ambiguous cases; `high+` avoids them.

## Prioritized next steps this evidence justifies

- **[done] Correlation-aware scoring** — pattern severity caps the trust score.
- **[next] Add a DNS-tunneling pattern detector** — reuse the indicators the
  `dnsHook` already produces; would catch the DNS-exfil class.
- **[next] Fix the obfuscation-scan race** — run/await the scan before report so
  `OBFUSCATION_DETECTED` reliably lands; consider an obfuscation+network pattern.
- **[config] Change the recommended/default gate to `high+`** and document the
  recall/FP tradeoff honestly.

## Reproduce

```
node benchmark/run.js          # prints the table above, writes results.json
```
